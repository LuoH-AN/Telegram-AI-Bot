# shell 工具归档

## 工具定位
`shell` 是旧工具系统里权限最高、能力最强、风险也最大的一个工具。它允许模型在容器内执行 shell 命令，但为了降低破坏性，围绕执行过程做了大量限制、清洗和状态持久化设计。

对应实现：`tools/shell.py`

## 对外暴露的函数
- 工具名：`shell_exec`
- 主要参数：
  - `command`
  - `timeout_seconds`
  - 其他与工作目录、环境恢复有关的参数

## 核心职责
1. 校验命令是否命中危险模式
2. 清洗环境变量，避免泄露敏感信息
3. 在受控工作目录中执行命令
4. 限制超时时间和输出长度
5. 对工作区和部分包管理状态做 HF dataset 备份/恢复

## 安全控制
### 1. 黑名单正则
`_BLOCKED_PATTERNS` 明确拦截：
- `rm -rf /`
- `shutdown/reboot/poweroff`
- `mkfs`
- 向 `/dev/*` 写入
- `sudo/su`
- fork bomb
- `mount/umount`
- `systemctl/service`
- 一些高危 `killall/pkill -9`

### 2. 环境清洗
`_clean_env()` 会移除包含以下关键词的环境变量：
- TOKEN
- SECRET
- KEY
- PASSWORD
- DATABASE_URL
- AUTH

然后只保留相对安全的 PATH/NVM 等运行环境。

### 3. 输出裁剪
输出使用 `_truncate_output()` 控制，最大 10000 字符，保留头尾片段。

## 持久化设计
这是该工具最特别的一点：它不只是“运行命令”，还试图让 shell 环境在多次调用之间保持连续。

### 工作区快照
会把工作区打包后同步到 HF dataset store：
- 按 user_id + cwd 做 workspace key
- 归档为 tar.gz
- 存 meta/index 信息

### 包环境恢复
还尝试恢复：
- apt 包
- pip 包
- npm 包
- 一些用户目录下的持久化路径

目标是让后续 `shell_exec` 能延续之前装好的依赖和状态。

## 设计特点
### 1. 不是简单 subprocess 包装
它本质上是一个“带状态恢复和安全约束的执行沙箱”。

### 2. 强依赖宿主环境
大量逻辑依赖：
- HF dataset store
- 容器文件系统
- 包管理器可用性
- shell 工具链

### 3. 风险高但能力极强
它能让模型直接操作文件、执行脚本、安装依赖，所以在整个旧工具系统里属于最高风险层级。

## 如果以后要恢复
最小恢复版本建议只保留：
1. 命令执行
2. 黑名单校验
3. 环境变量清洗
4. 超时和输出截断

HF 持久化、包恢复这些高级能力建议后置恢复，因为它们复杂度非常高。

## 恢复时建议
未来若重建，最好把 shell 分成两层：
- 安全执行层
- 状态持久化层

不要再把命令执行、工作区快照、包环境恢复、HF 备份全部耦合在一个超大工具文件里，否则维护成本会持续升高。
